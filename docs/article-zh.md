# Claude Code 源码泄露全面拆解：架构、隐藏功能与工程启示

Claude Code 是 Anthropic 推出的 AI 编程助手 —— 一个能读代码、跑命令、改文件、管理复杂工程任务的 CLI 工具。2025 年初，它的完整源码意外泄露。本文基于对泄露源码的逐文件分析，带你深入了解这个 50 万行代码背后的架构设计。

## 这次泄露是怎么发生的

和大多数现代 Web 应用一样，Claude Code 以压缩过的 JavaScript bundle 发布。但它的生产构建中包含了 **source map** 文件 —— 这是一种将压缩代码映射回原始 TypeScript 源码的调试辅助文件。

Source map 本身是标准开发工具，但只应出现在开发环境中。当它们随生产版本一起发布时，任何下载了 bundle 的人都能还原出完整的源文件树，包括注释、变量名和模块结构。这正是此次泄露的原因：source map 随压缩后的代码一起发布，社区很快提取了完整源码。

这并非新型攻击手段。Source map 泄露已经暴露过许多生产应用的内部实现。对工程团队的启示很明确：**务必从生产构建中剥离 source map**，或仅在经过身份验证的调试会话中提供。

## 泄露规模有多大

提取出的代码库规模相当可观：

| 指标 | 数值 |
|------|------|
| 源文件数（TS/TSX/JS/JSX） | ~1,900 个 |
| 代码总行数 | ~513,000 行 |
| 顶层模块数 | 36 个 |
| LLM 可调用工具数 | 39 个 |
| React Hooks | 85+ 个 |
| UI 组件 | 146+ 个 |
| 斜杠命令 | 100+ 个 |

架构覆盖面很广：终端 UI 渲染（自定义 Ink.js 分支）、LLM 编排引擎、带权限门控的完整工具系统、IDE 双向桥接协议、后台任务管理、持久化记忆、插件/技能扩展体系，以及多 Agent 协调机制。

## 架构深度拆解

### 工具系统：39 个工具的分类与权限模型

工具是 LLM 与外部世界交互的主要接口。每个工具通过 `buildTool()` 工厂函数（`/Tool.ts`）构建，遵循统一的接口契约：

- **输入/输出 Schema**（基于 Zod 验证）
- **权限检查**（`checkPermissions()` → 自动放行 / 询问用户 / 拒绝）
- **元数据标记**：`isReadOnly`（是否只读）、`isDestructive`（是否有破坏性）、`isConcurrencySafe`（是否并发安全）、`shouldDefer`（是否延迟加载）

39 个工具横跨 10 个类别：

| 类别 | 工具 | 说明 |
|------|------|------|
| 文件操作 | `Read`, `Edit`, `Write`, `Glob`, `Grep` | 读写文件、模式匹配、内容搜索 |
| 命令执行 | `Bash`, `PowerShell` | Shell 命令执行，带 AST 安全检查 |
| 多 Agent | `Agent`, `SendMessage`, `TeamCreate/Delete` | 子 Agent 生成、消息传递、团队管理 |
| 任务管理 | `TaskCreate/Get/List/Update/Stop` | 后台任务生命周期 |
| 网络访问 | `WebFetch`, `WebSearch` | 网页抓取、搜索 |
| MCP 集成 | `MCPTool`, `McpAuth`, `ListMcpResources` | Model Context Protocol 服务器交互 |
| 模式控制 | `EnterPlanMode`, `EnterWorktree`, `Config` | 计划模式、工作树隔离、配置管理 |

**权限模型是最值得研究的部分。** `Bash` 命令会被 **AST 解析**，判断是只读操作（自动放行）还是破坏性操作（需用户确认）。`WebFetch` 使用基于域名的规则，有预批准域名列表。部分工具根据输入内容动态计算权限级别。自动模式下，轻量级分类器提供安全操作的快速自动批准。

默认值采用 **fail-closed 策略**：`isReadOnly` 默认 `false`，`isConcurrencySafe` 默认 `false` —— 工具必须显式声明才能获得更宽松的权限。

### Query Engine：核心编排引擎

Claude Code 的核心是 `/query.ts` 中的 `queryLoop()` —— 一个异步生成器，实现了 Agent 式的多轮循环：

**第一步：预处理**
- 微压缩（microcompact）：移除冗余空白
- 自动压缩（autocompact）：超过阈值时自动总结
- 上下文折叠（context collapse）：激进的上下文窗口优化
- 工具结果预算：限制工具返回结果的大小

**第二步：流式 LLM 调用**
- `queryModelWithStreaming()` 带重试逻辑和回退模型
- 速率限制检测和 529 连续错误追踪
- 流式传输 token 给前端渲染

**第三步：流式工具执行（关键优化）**
- `StreamingToolExecutor` 在 LLM 还在生成时就开始执行并发安全的工具
- 显著降低多工具轮次的延迟

**第四步：后续工具执行**
- `toolOrchestration.ts` 将工具分为并发批次（只读）和串行批次（写操作）
- 只读工具并行执行，写工具逐个执行

**第五步：错误恢复**
- Prompt 过长（413）→ 上下文折叠 → 反应式压缩
- 输出 token 超限 → 升级到 64K → 多轮恢复（最多 3 次）
- 流式回退 → 丢弃部分结果，切换回退模型重试

**第六步：停止钩子 + 循环**
- 记忆提取、技能发现、任务完成评估
- 工具结果 + 附件消息（记忆、技能、命令）回传，循环继续

整个系统有 **7+ 个不同的续行路径**，处理了从模型回退到上下文溢出到 token 预算耗尽的各种场景。

### IDE 桥接：双向通信协议

`/bridge/` 模块实现了 CLI 与 IDE 扩展（VS Code、JetBrains）之间的双向通信。支持两种协议：

| 协议 | 入站（IDE→CLI） | 出站（CLI→IDE） | 选择依据 |
|------|----------------|----------------|----------|
| v1 | WebSocket | HTTP POST | 服务端按会话选择 |
| v2 | SSE 流 | HTTP POST via CCR | `use_code_sessions` 标志 |

关键设计细节：
- **工作轮询**：指数退避（2s → 120s），最长放弃时间 10 分钟
- **消息去重**：有界 UUID 集合（容量 2000），防止回声和重传
- **两种重连策略**：就地重用（环境仍存活）vs 全新会话（环境已过期）
- **心跳续租**：v2 协议 20s 心跳，工作项 60s 心跳
- **控制请求**：IDE/服务器可发送 `initialize`、`set_model`、`set_permission_mode`、`interrupt`，CLI 必须在 ~10-14s 内响应
- **崩溃恢复**：持久化桥接指针文件（`~/.claude/bridge-pointer-{dir}.json`），重启后尝试重连

## 隐藏功能揭秘

### BUDDY：确定性虚拟宠物系统

藏在 `/buddy/` 目录中，这是一个完整的虚拟宠物系统。每个用户根据 `userId` 获得唯一的 ASCII 艺术伙伴。

**生成算法：**
```
userId + SALT("friend-2026-401") → FNV-1a 哈希 → Mulberry32 PRNG 种子 → 确定性属性抽取
```

Mulberry32 是一个轻量级 32 位伪随机数生成器，相同种子永远产生相同序列。这意味着 **同一个用户永远得到同一个宠物**，无需服务器调用。

**18 个物种**：鸭子、鹅、果冻、猫、龙、章鱼、猫头鹰、企鹅、乌龟、蜗牛、幽灵、六角恐龙、水豚、仙人掌、机器人、兔子、蘑菇、肥墩。物种名以十六进制字符编码，规避 bundle 中的字符串扫描。

**5 个稀有度等级**：

| 稀有度 | 概率 | 属性下限 | 闪光概率 |
|--------|------|----------|----------|
| 普通 | 60% | 5 | 1%（独立） |
| 非凡 | 25% | 15 | 1% |
| 稀有 | 10% | 25 | 1% |
| 史诗 | 4% | 35 | 1% |
| 传说 | 1% | 50 | 1% |

**防作弊设计**：只有"灵魂"（名字、性格）持久化到配置文件，"骨骼"（物种、稀有度、属性）每次从 userId 哈希重新生成。即使手动编辑配置文件，也无法伪造传说级稀有度。

**视觉效果**：3 帧 ASCII 动画、帽子装饰（王冠、礼帽、螺旋桨帽等）、语音气泡、抚摸反应（爱心飘浮动画）。初始展示窗口为 2026 年 4 月 1-7 日（本地时区滚动触发）。

### KAIROS：主动感知助手

KAIROS（`/assistant/` + `/tools/BriefTool/`）从根本上改变了 Claude 的通信方式。

**核心机制**：所有可见输出必须通过 `SendUserMessage` 工具发送。工具调用之外的文本只进入低可见度的详情视图。这使得 Claude 能够：

1. **主动发送状态更新**（后台任务完成、发现阻塞）—— 使用 `status: 'proactive'`
2. **检查点式通信**：确认（一句话）→ 执行工作 → 发送结果，跳过无意义的中间更新
3. **追加写入日志**：持续助手会话使用按日命名的追加日志，夜间通过 `/dream` 技能蒸馏到 MEMORY.md

**多层门控**：构建时特性标志、服务端 GrowthBook 门控（每 5 分钟刷新，支持实时关闭）、用户显式启用。

## 值得深入研究的冷门模块

### coordinator/ — 多 Agent 协调

实现协调者模式：Claude 不直接操作代码，而是生成工作 Agent、分配任务、综合结果。系统提示词明确教导协调者"不要感谢 worker"、"不要不必要地串行化工作"、"综合发现而非转发"。Worker 获得过滤后的工具访问权限，无法使用团队管理工具。

### memdir/ — 持久化记忆

实现了完整的记忆系统（MEMORY.md 索引 + `~/.claude/memory/` 下的主题文件）。关键设计：

- **双级上限**：MEMORY.md 最多 200 行 / 25KB
- **Sonnet 驱动的相关性选择**：每次查询选出最相关的 5 个记忆文件
- **路径遍历防御**：拒绝 null 字节、URL 编码遍历、Unicode 标准化攻击
- **四种记忆类型**：`user`（用户画像）、`feedback`（行为指导）、`project`（项目上下文）、`reference`（外部资源指针）
- **明确排除可推导信息**：代码模式、git 历史、调试方案不存记忆 —— 只保存无法从项目中提取的上下文

### moreright/ — 神秘的内部功能占位

一个 25 行的 no-op 存根，替代了一个仅限 Anthropic 内部的功能。通过 `"external" === 'ant'`（永远为 `false` 的字符串比较）确保外部构建永远不会激活。

接口暴露了两个生命周期钩子：
- `onBeforeQuery()` —— 查询执行前（可拦截/修改输入）
- `onTurnComplete()` —— 查询完成后（清理/后处理）

这揭示了 Anthropic 的 **内部/外部构建分离策略**：同一代码库，通过永假门控 + 空实现，确保内部功能零泄露。

## 对 AI 工程师的启示

**1. 权限设计的复杂度可能超过 LLM 编排本身。** Claude Code 的权限系统（AST 解析命令、输入内容动态计算、域名规则、自动分类器、计划模式）投入了大量工程。如果你在构建让 LLM 执行代码的工具，这里值得重点投入。

**2. 流式工具执行是显著优化。** 在 LLM 还在生成时就开始执行并发安全的工具（`StreamingToolExecutor`），在多工具轮次中能明显降低延迟。这是一个容易被忽视但收益很大的优化点。

**3. 上下文管理是持续战役。** 代码库有四种不同的压缩策略（微压缩、自动压缩、反应式压缩、上下文折叠）加上工具结果预算。上下文窗口管理不是一个"解决了就完事"的问题，而是需要持续迭代的工程挑战。

**4. 内部功能的优雅隔离。** `moreright/` 的模式（永假字符串门控 + 空存根）是在单一代码库中维护内部功能的干净方案，值得借鉴。

**5. 确定性生成的巧妙应用。** BUDDY 的方法（哈希 → PRNG 种子 → 属性抽取）是为每个用户生成个性化内容的优雅模式 —— 无需服务器调用或持久化状态，只有模型生成的部分需要存储。

**6. Fail-closed 的默认值设计。** `buildTool()` 默认 `isReadOnly: false`、`isConcurrencySafe: false` —— 安全的选择是保守的选择。工具必须显式声明才能获得更宽松的行为，这比"默认开放，出问题再收紧"安全得多。
